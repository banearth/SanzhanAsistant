# 三战助手 MVP 功能清单

> 目标
>
> 第一版不是做“全自动 AI 指挥官”，而是做一个**稳定、省时间、可复跑**的三战日常助手。
>
> 成功标准不是“功能看起来很多”，而是：
>
> - 你可以一键启动
> - 它能稳定做完一轮高频日常
> - 出错时能停在可排查的位置
> - 你第二天能看懂它昨天做了什么

## 1. MVP 定义

本阶段的 MVP 定义如下：

- 聚焦 `日常自动化`
- 默认 `单账号、单模拟器、单分辨率`
- 默认 `低风险动作自动执行`
- 默认 `作战相关只做观察和预留，不做自主指挥`

不纳入本阶段的内容：

- 多账号调度
- 多分辨率适配
- 高风险作战自动决策
- 联盟级复杂协同
- 重型桌面 UI
- 端到端视觉大模型直接控屏

## 2. MVP 北极星指标

建议第一版就盯下面 5 个指标：

| 指标 | 目标 |
| --- | --- |
| 单轮日常成功率 | `>= 80%`，且失败可定位 |
| 单轮无人干预时长 | `15-30 分钟` 内可稳定运行 |
| 异常可恢复率 | 常见弹窗/断线/页面偏移可自动恢复一部分 |
| 日报可读性 | 你能在 1 分钟内看懂“做了什么、失败在哪里” |
| 新任务接入效率 | 新增 1 个简单日常任务不应改动太多核心代码 |

## 3. 功能优先级

建议按 `P0 / P1 / P2` 推进。

### P0：必须有

P0 是“没有就不算 MVP”的部分。

#### 3.1 运行与连接

- `P0` 模拟器连接
  - 能连接目标环境
  - 能截图
  - 能执行点击/滑动/返回
- `P0` 启动前检查
  - 检查 ADB/控制链路是否可用
  - 检查游戏是否已启动或可拉起
  - 检查配置是否完整

#### 3.2 页面识别与状态机

- `P0` 主页面识别
  - 主城
  - 世界地图
  - 任务页
  - 邮件页
  - 征兵/队列相关页面
- `P0` 弹窗识别
  - 奖励弹窗
  - 确认弹窗
  - 网络异常/重试
  - 未知弹窗兜底
- `P0` 页面回正能力
  - 如果不在预期页面，能回到主流程锚点

#### 3.3 基础日常任务

- `P0` 登录与重连
  - 进入游戏
  - 断线后尝试恢复
- `P0` 奖励领取
  - 可安全领取的通用奖励
- `P0` 邮件处理
  - 进入邮件页
  - 领取可领内容
- `P0` 常规任务处理
  - 进入任务入口
  - 领取已完成任务
- `P0` 基础资源循环
  - 征兵相关操作
  - 可低风险重复的资源产出/队列补充

#### 3.4 执行可观测性

- `P0` 结构化日志
  - 记录任务开始/结束/失败节点
- `P0` 关键截图保存
  - 失败时保存截图
  - 关键节点可选保存截图
- `P0` 执行摘要
  - 今天执行了哪些任务
  - 哪些成功
  - 哪些失败

#### 3.5 安全与风控

- `P0` 白名单动作
  - 只允许运行低风险动作
- `P0` 失败停止
  - 连续失败达到阈值后停止
- `P0` 手动接管预留
  - 出现未知状态时停止而不是乱点

### P1：应该有

P1 是“明显提升体验，但没有也能先用”的部分。

对三战这个项目来说，需要额外强调一点：

- `同盟任务/同盟目标` 不是普通附加功能，而是接近核心日常入口
- 后续的“日常自动化”不能只理解成领邮件、领奖励，还要覆盖“查看同盟目标并组织派兵”这类战斗型日常

#### 3.6 稳定性增强

- `P1` 页面偏移恢复
- `P1` 多路径回主页
- `P1` 重试与退避
- `P1` 超时治理
- `P1` 常见异常分类

#### 3.7 日常调度能力

- `P1` 任务优先级控制
  - 先领奖还是先补队列
- `P1` 条件执行
  - 资源不足时跳过某任务
  - 队列已满时跳过某任务
- `P1` 配置开关
  - 是否执行邮件
  - 是否执行征兵
  - 是否只做收菜

#### 3.8 报告增强

- `P1` 每日执行报告
  - 总耗时
  - 成功任务数
  - 失败任务数
  - 失败原因分类
- `P1` 最近一次状态快照
  - 资源概况
  - 队列概况
  - 异常摘要

#### 3.9 AI 轻量参与

- `P1` AI 日常排序建议
  - 基于状态快照给任务排序
- `P1` AI 异常说明
  - 把失败日志转成你能快速读懂的摘要

注意：

P1 的 AI 只做“建议和解释”，不直接接管底层控屏。

### P2：可后置

P2 是“重要，但不该阻塞 MVP”的部分。

#### 3.10 作战相关预埋

- `P2` 地图态识别增强
- `P2` 队伍状态读取
- `P2` 坐标读取与校验
- `P2` 作战建议 DSL
- `P2` 低风险派遣模板

#### 3.11 工程化增强

- `P2` Web/桌面控制台
- `P2` 多账号支持
- `P2` 任务编排可视化
- `P2` 历史数据统计看板

## 4. 推荐 MVP 任务包

为了尽快见效，我建议把 MVP 再压缩成下面 4 个任务包。

### 包 A：基础运行闭环

内容：

- 启动检查
- 连接模拟器
- 拉起游戏
- 识别主界面
- 执行一个简单点击任务

验收：

- 连续跑 10 次，至少大多数能到主界面并完成任务

### 包 B：日常主链路

内容：

- 登录/重连
- 邮件领取
- 任务领取
- 通用奖励领取
- 一项稳定的资源/队列补充动作

验收：

- 一键运行可完成一轮“有感知价值”的收菜流程

### 包 C：异常恢复

内容：

- 奖励弹窗处理
- 确认框处理
- 网络异常处理
- 页面走丢回正
- 未知状态停止并截图

验收：

- 出现常见异常时，不会直接卡死或误点失控

### 包 D：日志与日报

内容：

- 结构化日志
- 失败截图
- 运行摘要
- 每日汇总文本

验收：

- 你第二天打开报告就知道昨天跑得怎么样

## 5. 建议的 MVP 状态模型

第一版不用建得太复杂，但一定要统一命名。

推荐至少有这些状态：

```text
APP_LAUNCHING
LOGIN_FLOW
HOME_CITY
WORLD_MAP
TASK_CENTER
MAIL_CENTER
RECRUIT_PANEL
REWARD_POPUP
CONFIRM_POPUP
NETWORK_ERROR
UNKNOWN_STATE
```

推荐至少有这些执行结果：

```text
SUCCESS
FAILED_RETRYABLE
FAILED_NON_RETRYABLE
SKIPPED_BY_CONDITION
STOPPED_BY_SAFETY
```

## 6. 建议的 MVP 动作 DSL

即使第一版 AI 参与很少，也建议先把动作层定义出来，避免以后返工。

示例：

```json
{
  "intent": "daily.collect_mail_rewards",
  "params": {},
  "preconditions": ["MAIL_CENTER"],
  "risk_level": "low",
  "on_fail": "retry_then_stop"
}
```

再比如：

```json
{
  "intent": "daily.recruit_soldiers",
  "params": {
    "mode": "safe_minimum"
  },
  "preconditions": ["HOME_CITY"],
  "risk_level": "low",
  "on_fail": "capture_and_skip"
}
```

这样后面 AI 规划层只需要决定：

- 做不做
- 先做哪个
- 用哪个策略参数

而不需要知道“具体点哪个按钮”。

## 7. MVP 验收清单

下面这张清单建议作为第一阶段的验收门槛。

| 项目 | 验收要求 |
| --- | --- |
| 连接能力 | 能稳定连接目标模拟器并截图 |
| 页面识别 | 能识别主城、任务、邮件和常见弹窗 |
| 日常流程 | 能完成至少 3 个高频日常动作 |
| 异常处理 | 遇到常见弹窗能继续，未知状态能停下 |
| 日志 | 有结构化日志和失败截图 |
| 报告 | 能输出一份人能看懂的执行摘要 |
| 安全 | 默认只执行低风险动作 |

## 8. 本阶段最容易失控的点

MVP 最容易失败的，不是代码写不出来，而是范围失控。

建议严格避免：

- 今天想做日常，明天突然插入复杂作战
- 页面状态还没打牢，就开始接 LLM 自主规划
- 识别还不稳，就急着做多环境适配
- 任务还没跑通，就先做前端面板

一句话：

**先证明“稳定收菜”成立，再扩成“智能作战”。**

## 9. 我建议的 MVP 终点

我认为这版 MVP 到这里就可以算成功：

- 你配置一次后可以一键运行
- 它能完成一轮低风险日常
- 中途出问题会留下日志和截图
- 第二天你能看到一份简洁摘要

如果做到了这四点，这个项目就已经从“概念”进入“能长期打磨的产品雏形”了。
